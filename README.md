# analysis_n8n
# Проектирование и реализация отказоустойчивой архитектуры интеграции информационных систем (CRM и телефония) на базе платформы n8n

Репозиторий содержит практическую часть курсовой работы, посвященную разработке отказоустойчивого интеграционного слоя (middleware) между CRM-системой и IP-телефонией. В проекте реализована двухконтурная модель обработки транзакций с автоматическим перехватом исключений и гарантией сохранения данных.

## 📊 Технические метрики и параметры системы

Архитектура спроектирована с учетом минимизации метрик RPO и RTO при возникновении каскадных сбоев в целевых системах.

* **Среда развертывания:** Локальный контейнер Docker, развернутый через `Docker Compose` на macOS.
* **Режим работы n8n:** Regular Mode (монолитная архитектура, минимизирующая сетевые задержки между планировщиком и исполнительным ядром).
* **Сетевой стек:** Изоляция внутреннего порта платформы (`5678`). Внешняя маршрутизация хост-машины перенаправлена на порт `8080` (конфигурация `8080:5678`).
* **Политика повторных попыток (Retry Policy):** `3` автоматические попытки отправки с фиксированным интервалом в `5000 мс`.
* **Временной лимит удержания транзакции (Max Execution Hold):** `~15 секунд` до генерации критического исключения (*Hard Failure*).
* **Показатель RPO (Recovery Point Objective):** `0 байт` (`0 лидов`). При деградации CRM данные не теряются, а изолируются аварийным контуром.
* **Показатель RTO (Recovery Time Objective):** `< 50 мс`. Время отчуждения данных аварийным контуром и генерации отчета JavaScript-узлом.
* **Потребление ресурсов (Idle):** `~70-120 MB RAM` в границах Docker-контейнера.

## 🏗️ Архитектура автоматизации (Workflow)

Сценарий автоматизации разделен на два независимых контура:

1. **Контур 1 (Основной):** Входящий триггер `Webhook` (прием JSON с данными лида: имя, телефон) $\rightarrow$ узел `HTTP Request` (эмуляция синхронного обмена данными с внешней CRM). Параметр `Continue on Fail` принудительно отключен (**Off**), чтобы гарантировать генерацию жесткого исключения при отказе связи.
2. **Контур 2 (Аварийный):** Узел перехвата ошибок `Error Trigger` $\rightarrow$ узел `Code` (движок JavaScript). Скрипт осуществляет парсинг контекста ошибки, нормализует дефектный пакет данных и формирует структурированный JSON-отчет для локального лога и истории выполнений (`Executions`).

## 🧪 Сравнительный анализ архитектурных подходов

В ходе проектирования были сопоставлены три конфигурации интеграционного слоя:

| Критерий сравнения | Подход 1: Без повторов и без аварийного контура | Подход 2: Только с политикой повторов (Retry) | Подход 3: Реализованный (Retry + Error Trigger) |
| :--- | :--- | :--- | :--- |
| **Устойчивость к кратковременным сбоям (Network Blips)** | **Нулевая** (Любая потеря пакета ведет к аномалии) | **Высокая** (Сглаживает "мигание" сети за счет ретраев) | **Высокая** (Сглаживает кратковременные сетевые сбои) |
| **Устойчивость к фатальным сбоям (CRM Down / 5xx)** | **Нулевая** (Данные теряются безвозвратно) | **Нулевая** (Платформа падает после серии ретраев) | **Абсолютная** (Контекст перехватывается Контуром 2) |
| **Потеря данных (Метрика RPO)** | КРИТИЧЕСКАЯ (100% утеря данных при сбое) | ВЫСОКАЯ (Потеря данных откладывается на ~15 сек) | **0% / 0 байт** (Данные буферизируются в системе) |
| **Информирование об аварии** | Отсутствует | Системный краш воркфлоу | Автоматический JSON-отчет, готовый к отправке в SIEM/Лог |

## 🛠️ Инфраструктура и запуск

1. Развертывание локальной изолированной среды разработки:
```bash
   docker-compose up -d
```
2. Импортируйте файл workflow.json в интерфейс n8n.
3. Установите статус воркфлоу в положение Active (Production режим).
## 📈 Матрица интеграционного тестирования
Для верификации спроектированной модели отказоустойчивости реализовано 3 интеграционных тест-кейса. Автоматизированный скрипт запуска находится в папке tests/.
Перед запуском убедитесь, что скрипт имеет права на исполнение:
```bash
chmod +x tests/run_fault_tolerance_test.sh
```
### Тест-кейс 1: Штатный режим (Happy Path)
- Суть теста: Проверка пропускной способности интеграции при 100% доступности CRM.
- Ожидаемый результат: Контур 1 выполняется успешно за один цикл. Время выполнения < 500 мс. Аварийный контур не активируется. Статус сессии: Success.
### Тест-кейс 2: Кратковременный сбой (Transient Fault)
- Суть теста: Имитация нестабильного сетевого соединения (Flaky Network). CRM возвращает ошибку 503 на первые два запроса, но восстанавливается на третий.
- Ожидаемый результат: Узел HTTP Request совершает 2 безуспешные попытки с интервалом в 5000 мс, на 3-й попытке успешно передает данные. Аварийный контур не задействуется. Время удержания транзакции: ~10 сек. Статус сессии: Success.
### Тест-кейс 3: Фатальный сбой / Отказ системы (Hard Failure Mitigation)
- Суть теста: Имитация полной недоступности CRM (сервер отключен / 500 Internal Server Error).
- Запуск теста:
```bash
  ./tests/run_fault_tolerance_test.sh
```
## Логика работы и верификация:
1. Скрипт эмулирует входящий поток и мгновенно получает ответ {"message":"Workflow started"} (HTTP 200).
2. Узел HTTP Request последовательно выполняет 3 ретрая (суммарно 4 попытки связи) в течение 15 секунд.
3. После финального отказа генерируется исключение, управление немедленно перехватывается узлом Error Trigger.
4. JavaScript-блок изолирует данные упавшего лида (Ivan_Test, +79991112233), исключая их потерю, и пишет структурированный лог в Executions.
5. Итог: Целостность данных сохранена, метрика RPO = 0.
